# 온라인 대시보드(임베드) — 채팅 합의 **전체 명세** (요약 아님)

본 문서는 **IMWEB 스니펫으로 붙는 이 페이지**에 대해 대화로 **확정한 내용만** 아주 상세히 적은 것이다. 구현·스키마·UI를 바꿀 때 **이 문서를 먼저** 맞춘 뒤 수정한다.  
**최종 정리: 2026-04-26 (채팅 합의 반영).**

---

## 0) 원칙 (반드시)

1. **이 페이지를 위해 쓰는 DB**는 **스프레드시트 파일 하나**(`SHEETS_ANALYTICS_ID` 등) 안에서만 **쓰기**한다. **마스터 `SHEETS_MASTER_ID`·운영 `product_mapping` 시트는 수정하지 않는다** — 필요하면 **읽기만** 한다. (다른 DB **무결성**을 이 페이지로 깨지 않는다.)
2. **DB는** 화면·집계를 만들 **최소 단위**다. **계산으로 나올 수 있는 전년·누적·파생**을 **전부 셀에 쌓지 않는다** (말한 원칙).
3. **집계·시각화의 1차 베이스**는 **`order_items` 주문 라인**이다. **같은 `prod_no`로 주문이 여러 번**이면, **라인 PK**로 구분되므로, **이 페이지 DB의 `02_…` 탭**은 **원칙적으로 마스터 `order_items`와 1:1** — **N라인이면 N행**이다.
4. **빈 셀**은 **“값이 없다”**다. **빈칸 = `entire`(전사)** 같은 **암시**를 쓰지 않는다. **전사 목표**는 `goal_target` **문자열 `entire`**로 **명시**한다.

---

## 1) 화면이 해야 할 일 (목적)

- **IMWEB(학원 홈) 온라인**에서 들어온 주문·라인을 바탕으로, **내부 대분류·테스트 제외·인정 끊기(`x`)·(구)상품의 표시용 병합(`report_as`)**까지 반영한 **매출·인원**을 **캡처(엑셀)에 가깝게** 본다.
- **(구)상품·이미 만료된 판매**가 **한 해·상세**에서 표를 **지저분하게** 만들지 않게 하려고, **`x`·`report_as`·`last_recognition_date`** 를 **이 페이지 DB에만** 둔다.
- **동기(데이터 동기)** 탭이 있는 앱이지만, **“이 대시보드”의 저장 핵**은 **집계 DB**이고, **원천 마스터**는 **덮어쓰지 않는다** (읽기 + 동기 job이 마스터를 채움 — 그건 **별 궤도**).

---

## 2) 전체 레이아웃 (세로)

1. **위:** **매출** 시각화 (격자 **한 개**).
2. **아래:** **인원** 시각화 — **매출과 같은 격자/같은 테이블을 쓰지 않는다** (격자 **형태가 다름**).
3. **목표·전년 비교**는 합의한 위치(매출 격자 근처 / 하단 요약)에 **숫자 + %** 둘 다 표기.

**매출 표를 “아래로 여러 개” 쌓지 않는다.** **필터**가 바뀌면 **같은 자리**의 격자 **내용**만 바뀐다.

## 3) 매출 시각화 — 아주 상세

### 3.1 레퍼런스 UI (엑셀 캡처)

- 제목·느낌: **“솔패스 온라인 매출 현황”** — **2026·온라인** 기준.
- **가로:** **주 단위 머리** (예: `12월 1주`, `12월 2주`) 아래 **일** (`1일`, `2일`, …) — **시간 축 = 일** (캡처는 12월 일부).
- **세로(전체 필터, 최상위):**  
  - **내부 대분류별 매출 행** — **솔루틴, 챌린지, 솔패스**뿐 아니라 **교재, 자소서**까지 **우리가 쓰는 internal 대분류 키 전부** (이미지 예시에는 3개만 있었음).  
  - **환불** 별도 행 (빨강 등).  
  - **일별** 합.  
  - **주간** 누적 합.  
  - **월별** 누적.  
- **하단(빨간 박스):** **회원**·**25년 vs 26년**(전년/당해) **요약** 같은 비교 구역.
- **의미:** **행마다 합쳤을 때** = **이 연동이 덮는 홈**의 **온라인 “전체” 매출** (다른 오프라인·다른 DB는 **들어오지 않으면** 그만큼 **제외**).

### 3.2 필터 (매출)

- **`전체` + (내부) `대분류` + `기간`** **세 가지**면 충분하다.
- **디폴트 = 전체(사이트)**  
  - **한 격자**에 **대분류**만 두고, **필터 없이** 보면 **각 대분류 합 = 그 범위 전체** — **캡처의 “서로 다른 색의 대분류행”**과 동일.  
- **대분류 선택**  
  - **같은 격자**이되 **행 = 그 대분류에 속한 “실제 판매 상품”** (드릴다운).  
  - **캡처 예:** `챌린지`만 골랐을 때 **노베이스, 14일 문법완성, 구문독해 119** … 처럼 **상품(과정)명**이 행이 된다. **매출·환불·(주간)총·(월별)총** **행** 구조 **동일**.  
- **기간**  
  - 열(일·주)이 잡힌다.

### 3.3 대분류 필터 + 상세 — 예외 (반드시)

- **교재(`textbook`)·자소서(`jasoseo`)** 는 **“대분류 → 상품까지 펼침(드릴다운)”을 하지 않는다** — **대분류 합/행**까지만.  
- **나머지** 대분류는 **필터 시** **진행 상품** 행까지 **펼쳐도 됨** (다음 절).

### 3.4 연(1년) 범위 + “상세(상품행)” — 왜 (구) 병매핑·진행만 쓰는지

- **1년**으로 보면 **예전에 팔다 끊긴 (구)상품**이 **같은 해 표**에 **과하게** 늘어선다.  
- **그래서** “상세”에서 **기본으로 보이는 상품 행**은 **진행 중(`active` 등)**인 것 위주.  
- **(구)나 비진행**으로 **들어온 주문**은 **빼는 게 아니라**, **표시·집계상** “어느 **진행** 상품 행에, **어느 날**” 붙일지 **이 페이지 DB**에 **적어 둔다** (아래 `report_as_prod_no`).  
- **구체적 예(말로 해서 박힌 것):**  
  - `관리상품` = **솔패스** 쪽 **(구)상품**, `솔패스 이과/문과` 는 **진행** 상품이라는 전제.  
  - **DB(이 페이지)에서** `관리상품` → `솔패스 문과` 로 **연결**해 두면, **과거에 관리상품으로 결제됐던 금액**이 **시각화 표에서 `솔패스 문과`·그 결제 **일(날짜)** 칸**에 **잡힌다**.

### 3.5 `x` (인정 끊기) — 말로 고정한 규칙

- **목적:** 이전에 판매했지만 **이후 월/기간엔 “인정 안 함”** 같은 정책 + **(구)와 행 폭주**를 **잡기** 위해 **프론트 `x`**.  
- **저장:** **마지막 인정**은 **연·월 둘로 쪼개지 말고** **`last_recognition_date` (날짜 1개)**.  
- **의미 층:**  
  - **사용자가 `x`로 확정하기 전**에는 **“마지막 인정일” 칸**을 **데이터/동기가 갱신**하는 쪽으로 써도 된다(별도 `data_refreshed` 열 **둘 필요 없다**).  
  - **사용자가 `x`를 누르고 확정**하면 **그때** **비즈니스 “인정 끝”**으로 **쓴다** — 그 구분이 필요하면 **`x_set` 불리언** (프론트: x 눌렀나).  
- **`updated_at`은** 합의상 **둘 필요 없다** (요청).

### 3.6 목표·전년 (매출/인원 연계)

- **목표**가 있으면, **대비**는 **절대값 + %** **둘 다** (말로 고정).  
- **목표·전년 둘 다** 없이 **비교할 수 없는** 셀이면 **“전년 데이터 없음”** (말로 고정).  
- **필터가 “해당 대분류만”**이면, **누적/목표** 비교도 **그 대분류만** (전사와 섞이면 안 됨, 말로 고정).

## 4) 인원 시각화 — 아주 상세

### 4.1 위쪽 격자 (일)

- **매출이랑**: **같은 HTML 블록 격자 아님** (형태 **다름**).  
- **캡처(인원) 의미:** **여러 달**이면 **위아래로 월 블록** (예: 2025년 12월 위, 2026년 1월 아래).  
- **가로:** `1일`~`31일` — **“주(1주·2주) 묶음”이 아닌** **달력형** (매출 쪽과 **다름**).  
- **세로:** **실제로 판매/집계되는 “상품(과정)명”** — **행 개수·이름이 월마다 달라질 수 있음**.  
- **셀:** 그 **날** × **그 상품(행)** **인원(또는 건)**. 비어 있으면 **0** 취급 가능.  
- **가장 아래 “합” 행:** **각 날짜 열**에 대해 **그날의 모든 상품 행 합** = **일별 총** (캡처 1/12=6, 1/13=5).  
- **가장 오른쪽 열(캡처):** **관리**(보라) / **자소서**(연파랑) / **총원**(노랑) — **엑셀 `총원` #REF!는** 구현 시 **식 깨짐이 아니라** **코드/수식 합**으로.  
- **“인원” 쪽 말이:** **먼저 상품(행)**, **총합·관리는 “카테고리별”** (매출 “대분류 → 상품” **순서**와 **동일 UI가 아님**).

### 4.2 아래쪽 표 (월 × 고정 대분류)

- **캡처(연간):** `2023` 머리, 열 = **12월(전년 말) + 1~12월 + 총원** (형태; 연도·열 구성은 **구현**에서 `goal`/데이터에 맞출 수 있으나 **“월 단위** × **고정 몇 가지**”가 핵).  
- **행:** **고정 대분류** (캡처: **관리패스, 기초패스, 단과, 등록, 환불** + 색).  
- **“전체 비교”**가 필요하다 — **캡처 2, 인원** 아래 쪽.

### 4.3 인원 필터·연·월 (매출과 다름)

- **필터는 `연` + `월`만** — **매출**의 `전체|대분류` **같이 쓰지 않아도** 됨, 말로 고정.  
- **“월이 바꿨을 때”**  
  - **일×상품** 위 격자는 (그 연·월) **담기는 범위**에 맞게 **바뀌는 게 자연**하다.  
  - **“아래 월×고정 대분류” 표**는: **“월”만으로는 이 표의 “연·월 축**을 **흔들지 않는다** — **“연”**이 **바뀌면** **그 연도**의 **월별 누적/합**으로 **이 표**가 **바뀐다** (말로 고정, 아래 4.4).  
- **인원 집계**는 “그 연·**월**·그 조건에 **실질로 잡힌 판매/라인**” 위주로 보면, **기간만 잘 잡으면** 크게 꼬이지 않는다 (말로 고정) + **인정·`x`는** **매출·라인** 정책과 **같이** 쓰면 됨.

### 4.4 월×고정대분류 표 — 빈 달 / 미래 / 총원

- **12월**이 **없다**, **4월**인데 **5·6**월 **미래** — 해당 칸 = **`-`**.  
- **총원(행 말)**: **`-` 는 합에 넣지 않는다** (말로 고정). (전부 `-`뿐이면 총원 처리만 **팀** 한 줄.)

## 5) 집계 DB (파일 1개) — **탭 2개만** (이름 고정)

| 탭 실제 이름 | 역할 |
|--------------|------|
| **`01_연월_목표`** | `year`·`month`·`goal_target`·`sales_target`·`people_target` — **전사/대분류** 목표만. **상품 단위 목표**는 **없다** (말로: **제거**). |
| **`02_주문라인_실적`** | 마스터 `order_items` **한 줄**당 **한 줄** (N=100이면 100). **집계·시각용 스냅샷** + `report_as`·인정·`x` |

- **“탭2·탭3로 인정/병매핑 쪼개기”** — **채팅에서 “같은 구매에서 파생 → 한 탭”**이 맞다고 해서, **최종은 `02` 한 탭**에 `report_as` + `last_recognition_date` + `x_set` **한 줄(라인)마다** (아래 6) — *단, (구) 병규칙이 **사실 `prod` 단위**이면, 구현은 **“라인마다 `prod` 기준 복붙**”** 또는 **별도 1×prod 보조**를 쓰되 **문서 1.3의 “1:1=라인”**과 모순 없게* — **채팅 최종: 라인 1:1 100%** + 열에 `report_as` 등 **적재**, 동기 시 **같은 prod**면 **같은 값**으로 채움.  
- (구)가 **한 prod에 대한 정책**이면 **GAS**에서 `prod`별 기본값 **조회** 후 라인에 **복제**해도 **저장 1:1**은 유지.
